UHS · Phase 6 · v1.0 · Handoff document

Wiring the prototype to a real product

Everything the dev needs to take the static HTML in site/ and admin/ and turn it into a multi-tenant SaaS on Supabase. This document is opinionated where it matters (RLS, the markup model, pricing visibility) and loose where the team should decide (framework specifics, CI).

01 · Overview

Recommended stack

The design assumes a thin frontend over Supabase. No bespoke API server is required for v1 — Supabase RLS + edge functions cover the surface.

Frontend
Next.js 14 (App Router) on Vercel — server components for the public site (SEO), client components for admin. Tailwind for production styling; the prototype CSS in design-system/ is the source of truth for tokens.
Backend
Supabase — Postgres, Auth, Storage, Edge Functions, Realtime.
Email
SendGrid (transactional). Webhooks back into Supabase for lead-thread sync.
SMS
Twilio. Same pattern — webhook → lead_messages insert.
Search
Postgres FTS for v1. Move to Typesense if inventory exceeds ~10k rows or filters get complex.
Analytics
PostHog (product) + GA4 (marketing site). Per-tenant dashboards via PostHog cohorts on org_id.

Why this stack and not Rails / Django

Three reasons specific to UHS:

  1. Multi-tenant from day one. Supabase RLS enforces tenant isolation in the database — not the app — which is harder to bypass and easier to audit. This matters more than usual because the white-label roadmap means we'll have lookalike orgs sharing infrastructure.
  2. Public site is content-heavy. Next.js + Vercel ISR gets us fast SEO-friendly listing pages without a CMS layer. The dealer admin is Supabase-driven; the public site reads from the same tables via the public anon key + read-only RLS policies.
  3. Lead inbox needs realtime. Supabase Realtime is free out of the box. New leads appear in the inbox without polling. SMS replies via Twilio inserts → realtime broadcast → toast in the dealer's browser.
Note for the dev: the prototype is HTML-only. There's no React component library to lift wholesale. Treat the prototype as a high-fidelity spec — copy markup and CSS class names, but rebuild the JSX in your component library of choice. The CSS in design-system/admin.css and design-system/tokens.css is portable as-is.

Row-level security — the non-negotiables

Every tenant-scoped table gets the same policy template. Helper:

-- Cached at session level so RLS doesn't re-query on every row
create function auth.org_ids() returns uuid[]
language sql stable security definer as $$
  select array_agg(org_id) from org_members
  where user_id = auth.uid() and status = 'active';
$$;

-- Applied to homes, leads, lead_messages, quotes, trade_ins, lots, etc.
create policy "tenant_isolation_select" on homes
  for select using (org_id = any(auth.org_ids()));

create policy "tenant_isolation_modify" on homes
  for all using (org_id = any(auth.org_ids()))
  with check (org_id = any(auth.org_ids()));

Two extras layered on top:

  • Role gating — separate policy that checks role for write operations on sensitive tables (e.g. only owner/manager can write to homes.base_price_cents and markup_pct).
  • Lot scoping — when org_members.scoped_lots is non-null, the user only sees homes/leads where lot_id = ANY(scoped_lots). Lexington-only sales staff don't see the Anderson pipeline.
Critical: the public site reads homes using the Supabase anon key. There must be a separate read-only policy that exposes only status = 'published' rows and excludes base_price_cents and markup_pct. Use a Postgres view (public_homes) and grant the anon role select access only to the view — never to the underlying table.
03 · The markup model

Why pricing has its own section

The prototype's defining feature: dealers think in base price + markup percentage, not in retail prices. The customer only ever sees the listed price. Three things must be enforced in code, not just in policy:

  1. Generated column. listed_price_cents is computed by the database, not the client. This prevents drift if someone updates the markup % but forgets to recompute. 
    listed_price_cents int generated always as (
  (base_price_cents * (100 + markup_pct))::int / 100
  + addons_cents
  + case when include_setup_in_price then setup_cents else 0 end
) stored
  2. Public view excludes internal fields. Use a view, not a column-level policy — column policies are a footgun: 
    create view public_homes as
  select id, org_id, lot_id, stock_no, name, manufacturer_id,
         model, type, beds, baths, sqft, listed_price_cents,
         status, on_lot_since, created_at
  from homes where status = 'published';
grant select on public_homes to anon;
  3. Quotes snapshot the price. When a quote is created, copy listed_price_cents into quotes.listed_price_cents. If the dealer later raises the markup %, outstanding quotes hold the original price until they expire.
White-label note: the default_markup_pct on orgs means each tenant carries their own pricing rule. When this becomes a product sold to other dealers, the markup % is a per-org config — not a hardcoded constant.

05 · Auth & org switching

How a session resolves to a tenant

  1. User logs in → Supabase issues JWT (claims: sub = user_id).
  2. Frontend calls org_members for that user. If 1 result → auto-select. If >1 → show org-switcher.html.
  3. Selected org_id is stored in: (a) localStorage for client-side gating, (b) a x-active-org header sent on every PostgREST request, (c) Postgres session variable via set_config('request.active_org', org_id) for use in policies that need which of the user's orgs is currently active.
  4. Switching orgs is just step 3 again. No re-login.
Why both auth.org_ids() AND an active-org header? The function returns all orgs the user belongs to (used by RLS for legitimate-access checks). The header narrows queries to the currently-viewed org so you don't accidentally show Lexington homes when the user is in the Anderson tab. Defense in depth.
06 · Leads pipeline

End-to-end flow

01

Customer fills the quote form on a home detail page.

Public site (anon key) POSTs to /functions/v1/lead-intake. The fn validates, inserts leads + initial lead_messages row, fires a SendGrid notification to the org's lead inbox address, and returns <200ms.

02

Realtime subscription pushes the new lead into the dealer's open inbox tab.

Subscribe pattern: supabase.channel('leads:'+orgId).on('postgres_changes', {event:'INSERT', schema:'public', table:'leads', filter:'org_id=eq.'+orgId}, …).

03

Salesperson replies via email or SMS from the lead detail page.

Insert into lead_messages with kind='outbound'. A trigger calls the appropriate edge function to actually send via SendGrid/Twilio. Reply-To uses a per-thread token so inbound replies route back to the right lead.

04

Customer replies. Inbound email/SMS hits webhook → edge fn → lead_messages insert with kind='inbound' → realtime broadcast → toast in dealer's tab.

05

Salesperson sends a quote.

Builds quote via /functions/v1/quote-build, which snapshots listed_price_cents into quotes, generates a PDF, stores it in Storage, returns a signed public URL. The lead's stage advances to 'quoted'.

06

Closed-won → homes.status flips to 'sold', lead moves to closed.

10 · Cutover roadmap

Suggested build order

Don't build everything at once. This sequence gets a working dealer flow live in ~6 weeks, then layers features. Each milestone is shippable to Marlena for real use.

WkMilestoneWhat ships
1FoundationSupabase project, schema migrations, RLS policies, auth, org switcher
2Inventory CRUDAdmin: list + edit + photos. Markup model live. Public site list page reads from public_homes.
3Public site (read)Home, inventory, home detail, financing static pages, contact form.
4LeadsQuote form intake, lead inbox, realtime, email reply (outbound only).
5Two-way commsInbound email parse, SMS via Twilio, internal notes, system events.
6Quotes & trade-insQuote PDF generation, public quote page, trade-in form & admin pickup.
7+PolishUsers & roles UI, audit log, financing pre-qual flow, analytics dashboards.

What to defer past v1

  • White-label. The schema supports it (per-org branding, default markup %, custom domain field) but don't build the multi-domain routing or per-tenant theming until the second dealer signs.
  • Financing pre-qual integration. Vanderbilt & 21st Mortgage have APIs, but partner integration is ~3 weeks each. Start with a static partner list + redirect-out flow.
  • Native apps. The lead inbox is the only screen with real native-app value (push notifications). Web push gets you 80% there for free.
11 · Open questions for product

Things the prototype assumes — confirm before coding

  1. Pricing visibility default. Prototype shows listed price publicly, base + markup never. Confirm this is the policy across all surfaces (site, quotes, embeds).
  2. Lead claim model. Are new leads round-robin, all-visible-claim-first, or manually assigned by manager? Prototype shows "unassigned" + manual assignment. Round-robin is a one-table change but needs a decision.
  3. SMS opt-in. TCPA compliance — do customers explicitly opt into SMS, or is the quote-form submission sufficient consent? Likely needs a checkbox.
  4. Financing pre-qual storage. Will pre-qual results ever land in our DB, or are they always external (link out, lender's site)? Affects whether we need PII handling.
  5. Audit retention. Manufactured-home sales have specific record-retention rules in SC (7 years for some docs). Need to confirm what we're storing on our side vs. the lender/state.
  6. Multi-org users. Prototype supports a user being in multiple orgs (Marlena could become a head-office user across white-label dealers). For v1, do we constrain to one-org-per-user, or build the switcher now?
That's the package. Pair this with the prototype's site/ and admin/ folders, the visual overviews (site/00-overview.html, admin/00-overview.html), and the brand guide (brand/00-brand.html). Open questions above should be the dev's first conversation with the product team — they unblock the schema and the lead flow.
